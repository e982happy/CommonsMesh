# 增量同步协议规范 (Incremental Synchronization Protocol)

在 CommonsMesh 的本地优先架构中，节点（移动端设备或桌面客户端）经常处于离线状态，或者仅能通过不可靠的 P2P 网络（如蓝牙、Wi-Fi Direct 或间歇性互联网连接）与其他节点通信。为了确保社区图谱数据的一致性，本协议设计了高效、低带宽的增量同步机制。

## 1. 核心目标

- **最小化带宽消耗**: 仅传输节点间缺失的增量事件（Delta Events），避免全量数据传输。
- **离线容忍**: 支持节点在断网状态下持续产生本地事件，并在重新连接时自动与网络中的其他节点合并。
- **去中心化合并**: 无需依赖中央服务器进行冲突解决，利用因果哈希链和 CRDT（Conflict-free Replicated Data Type）原则保证最终一致性。

## 2. 同步流程

节点之间的同步分为以下几个阶段：发现（Discovery）、协商（Negotiation）、传输（Transfer）与合并（Merge）。

### 2.1 发现 (Discovery)

节点通过 libp2p 的多种发现机制（如 mDNS 局域网发现、Kademlia DHT 全局发现）找到潜在的同步对等点（Peers）。一旦建立连接，双方会交换基础元数据（如协议版本、支持的同步主题）。

### 2.2 协商 (Negotiation)

协商阶段的核心是确定双方事件日志的差异（Set Reconciliation）。

#### 2.2.1 同步请求 (Sync Request)

发起方节点向目标节点发送 `sync.request` 消息。该消息的 Payload 包含发起方当前已知的事件摘要信息，通常采用以下几种方式之一：
- **Bloom Filter / Cuckoo Filter**: 发起方将其本地事件的哈希值编码为布隆过滤器或布谷鸟过滤器发送给目标节点。目标节点通过查询该过滤器，快速识别出自己拥有但发起方缺失的事件。
- **Merkle Search Tree (MST)**: 对于数据量较大的图谱，节点可维护一棵 Merkle 搜索树。双方通过比较树根哈希，逐步向下层请求不匹配的节点，精确定位缺失事件。
- **最新哈希链指针 (Latest Hash Pointers)**: 针对特定的项目或主题，发起方发送其已知的最新事件哈希。目标节点从该哈希开始，回溯并收集后续的所有增量事件。

#### 2.2.2 摘要比较

目标节点接收到 `sync.request` 后，根据 Payload 中的摘要信息，计算出需要发送给发起方的事件列表（Delta Set）。

### 2.3 传输 (Transfer)

目标节点将计算出的 Delta Set 打包，通过 `sync.snapshot` 消息发送给发起方。

#### 2.3.1 增量快照 (Sync Snapshot)

`sync.snapshot` 消息的 Payload 包含一个按因果顺序排列的事件数组。为了优化传输效率，事件可进行批量压缩（如 Gzip 或 Zstandard）。
- **分块传输**: 如果缺失事件数量庞大，目标节点可将快照分块（Chunks），并通过多次 `sync.snapshot` 消息发送，支持断点续传。

### 2.4 合并 (Merge)

发起方接收到 `sync.snapshot` 后，开始本地合并过程。

#### 2.4.1 验证与追加

1. **解包与解析**: 提取 Payload 中的事件数组。
2. **逐一验证**: 对每个事件执行核心引擎的 `validateAndAuthorize` 流程，包括哈希完整性、签名、能力令牌、防重放等检查。
3. **因果排序**: 根据事件的 `prev_event_hash` 确保事件按正确的因果顺序被处理。如果发现缺失的前驱事件（Causal Break），则将当前事件放入挂起队列（Pending Queue），并向目标节点发起针对缺失前驱的补充 `sync.request`。
4. **追加日志**: 验证通过的事件被追加到本地事件日志（Event Log）中。

#### 2.4.2 状态物化 (State Materialization)

新追加的事件会触发核心引擎的 `applyMessage` 逻辑，更新本地的图谱数据库（Graph Database）和各类业务状态（项目、任务、选举等）。由于所有事件都是不可变的（Append-Only），且遵循确定的状态转移规则，不同节点即使以不同顺序接收到并发事件，最终物化出的状态也将保持一致。

## 3. 冲突解决与 CRDT

在去中心化网络中，并发操作不可避免（例如，两名用户同时对同一个任务发起 `task.commit`）。

- **最后写入胜出 (Last-Write-Wins, LWW)**: 对于简单的属性更新（如用户个人资料），可采用基于事件时间戳（`created_at`）或逻辑时钟（Logical Clock）的 LWW 策略。
- **多值寄存器 (Multi-Value Register)**: 对于可能产生冲突的状态（如任务分配），系统保留所有并发的有效事件。应用层（如 UI 或 LLM 匹配引擎）负责向用户展示冲突状态，并提供解决机制（如发起 `dispute.report` 或由项目协调人发布新的决定性事件）。

## 4. 隐私与选择性同步

为了保护用户隐私，节点可配置同步策略：
- **主题订阅 (Topic Subscriptions)**: 节点仅请求和接收与其相关的项目或主题的事件。
- **选择性披露**: 敏感信息（如精确位置、个人联系方式）在事件 Payload 中加密，仅对持有特定解密密钥（如项目成员）的节点可见。未授权节点仍可同步和转发该事件（作为不透明的密文），以协助网络路由，但无法读取内容。

通过这种灵活、去中心化的增量同步机制，CommonsMesh 确保了即使在极度碎片化的网络环境中，社区成员也能保持高效的信息流通和协作。
